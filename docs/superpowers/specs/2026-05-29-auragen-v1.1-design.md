# Auragen v1.1 — Design Spec

**Status:** approved design, ready for implementation plan
**Date:** 2026-05-29
**Owner:** Fernando
**Builds on:** v1 spec at `docs/superpowers/specs/2026-05-28-auragen-design.md`

## 1. Purpose

Three additions on top of the working v1, based on feedback after using it in the browser:

1. **Edge hardness slider** — at v1's `blur=0`, the gradient itself is too soft. Need an independent control that hardens the blob edge (compresses gradient stops toward the center).
2. **Irregular blob shapes** — v1's blobs are perfect circles. Need organic, irregular silhouettes without leaving Canvas 2D.
3. **Custom palette** — let the user define their own palette (5 colors: background + 4 blob gradient stops), alongside the 3 presets.

All three are additive — nothing in v1 is removed or breaks. The render pipeline, file structure, and architecture stay intact.

## 2. Scope

### In scope (v1.1)

- New `hardness` field on `RenderParams` (0–1), surfaced as a sidebar slider
- New `irregularity` field on `RenderParams` (0–1), surfaced as a sidebar slider
- New `'custom'` palette: 4th chip in the palette selector, expands to show 5 color pickers
- `applyHardness()` helper in `src/renderer/render.ts` (pure)
- Cluster-jitter logic in `src/renderer/composition.ts` (pure, uses existing PRNG)
- `buildCustomPalette()` in `src/data/palettes.ts` (pure)
- Tests for new pure functions

### Out of scope (deferred to v2+)

- Manual blob positioning (drag on canvas)
- Per-blob color overrides (each blob a different palette variant)
- Multi-frequency / color / film-grain texture (still the v1 minimum grain)
- WebGL renderer for true metaball shapes
- Saving / loading / sharing compositions
- Custom palette persistence across sessions

## 3. Architecture

No structural changes. The renderer stays pure, decoupled from React. The 3 features plug into the existing extension points:

- Hardness → transforms gradient stops at draw time inside `drawBlob`
- Irregularity → expands each logical blob into a cluster of mini-blobs inside `generateComposition`
- Custom palette → built on demand from user-picked colors via `buildCustomPalette`, then flows through the existing `Palette` contract

The renderer doesn't know whether a palette is preset or custom — both are `Palette` objects.

## 4. Feature 1 — Edge Hardness Slider

### UI

New slider in the sidebar, below `Blur`:

| Label  | Range | Step | Default |
|--------|-------|------|---------|
| Dureza | 0–1   | 0.01 | 0       |

At 0, behavior is identical to v1. Increasing values progressively harden the blob edge.

### Renderer change

In `src/renderer/render.ts`, add a pure helper:

```ts
export function applyHardness(stops: GradientStop[], hardness: number): GradientStop[] {
  const factor = 1 - hardness;
  return stops.map((s) => ({ ...s, offset: s.offset * factor }));
}
```

In `drawBlob`, transform the variant's stops before adding them to the gradient:

```ts
const transformedStops = applyHardness(variant.stops, params.hardness);
for (const stop of transformedStops) {
  grad.addColorStop(stop.offset, rgba(stop.color, stop.alpha));
}
```

### How it produces hard edges

The presets (Nightfall/Eclipse/Aurora) all end with an `alpha: 0` stop at offset 1.0. When hardness compresses offsets toward 0, that transparent stop moves inward — e.g. with hardness=0.5, Nightfall's stops collapse from `[0, 0.35, 0.65, 1.0]` to `[0, 0.175, 0.325, 0.5]`. The outer half of the bounding box renders transparent. Visually: the blob is smaller AND has a sharper transition to nothing.

At hardness=1, all stops collapse to offset 0 and the blob effectively vanishes. In practice users will stay in the 0–0.8 range.

### Tests

Add to a new file `tests/renderer/render-helpers.test.ts`:
- `applyHardness` with hardness=0 returns identical stops (identity)
- `applyHardness` with hardness=0.5 halves all offsets
- `applyHardness` preserves color and alpha (only offsets change)

## 5. Feature 2 — Irregularity (Cluster Jitter)

### UI

New slider in the sidebar, between `Nº de blobs` and `Grão`:

| Label          | Range | Step | Default |
|----------------|-------|------|---------|
| Irregularidade | 0–1   | 0.01 | 0       |

At 0, behavior is identical to v1 (1 circle per logical blob). Increasing values jitter each logical blob into 4 mini-blobs.

### Composition change

`generateComposition` signature gains a parameter:

```ts
generateComposition(
  seed: number,
  blobCount: number,
  palette: Palette,
  irregularity: number,
): Composition
```

For each logical blob (blobCount iterations):
- Pick center `(cx, cy)`, base radius `R`, and `variantIdx` from PRNG as today
- If `irregularity === 0`: emit a single `CompositionBlob` at that position (current behavior)
- Else: emit 4 mini-blobs, each with:
  - Position: `(cx + jx, cy + jy)` where `jx, jy = (rng() - 0.5) * 2 * R * 0.6 * irregularity`
  - Radius: `R * (0.5 + rng() * 0.3)` — between 50% and 80% of base
  - Same `variantIdx`

Number of mini-blobs is fixed at 4. Spread scales with `R * 0.6 * irregularity` so clusters stay compact at low irregularity and become more rasgada at high irregularity.

Total RNG calls per logical blob:
- irregularity=0: 4 (same as v1: x, y, radius, variantIdx)
- irregularity>0: 4 (cx, cy, R, variantIdx) + 4 × 3 (per mini: jx, jy, miniR) = 16

Different call counts at different irregularity values change downstream sequences. This means the same seed produces different compositions when irregularity is toggled — acceptable, since toggling irregularity is itself a user-driven re-randomization.

### Render change

None. The renderer just draws whatever blobs are in the composition. With `globalCompositeOperation = 'lighter'` and the existing blur, the 4 mini-blobs in a cluster melt into a single irregular silhouette.

### Tests

Update `tests/renderer/composition.test.ts`:
- New test: with irregularity=0, composition has exactly `blobCount` blobs (unchanged)
- New test: with irregularity>0, composition has `blobCount * 4` blobs
- New test: with irregularity>0, blobs cluster — at least one pair of blobs is within `R` of each other for each logical blob (statistical check)
- New test: deterministic — same seed + irregularity → identical composition

## 6. Feature 3 — Custom Palette

### UI

The palette section gets a 4th chip: **Custom**. Layout:

```
Paleta
[ Nightfall ] [ Eclipse ] [ Aurora ] [ Custom ]
```

When `Custom` is selected, the section expands below the chips to show 5 color pickers stacked vertically:

```
Fundo    [#1a0d3d] [▦]
Centro   [#ff7a4d] [▦]
Anel 1   [#ec4899] [▦]
Anel 2   [#6b46c1] [▦]
Borda    [#000000] [▦]
```

`[▦]` is an HTML5 `<input type="color">`. The hex text is for reference (read-only label). Native picker, no library dependencies.

Default colors when first selecting Custom: Nightfall's values (background + the 4 stop colors). User can change any; changes update the preview live.

### Data model

`buildCustomPalette` lives in `src/data/palettes.ts`:

```ts
export function buildCustomPalette(
  colors: [string, string, string, string, string],
): Palette {
  return {
    id: 'custom',
    name: 'Custom',
    background: colors[0],
    blobVariants: [{
      stops: [
        { offset: 0,    color: colors[1], alpha: 1.0  },
        { offset: 0.33, color: colors[2], alpha: 0.85 },
        { offset: 0.66, color: colors[3], alpha: 0.5  },
        { offset: 1,    color: colors[4], alpha: 0    },
      ],
    }],
  };
}
```

The structure mirrors the presets exactly (1 variant, 4 stops, transparent edge). The hardness slider works on Custom palettes the same way it works on presets.

The "Borda" color has `alpha: 0` but still influences interpolation — the radial gradient interpolates color AND alpha between stops, so the Borda hue contributes to the transition between Anel 2 and edge.

### State in App.tsx

```ts
const [customColors, setCustomColors] = useState<
  [string, string, string, string, string]
>(['#1a0d3d', '#ff7a4d', '#ec4899', '#6b46c1', '#000000']);
```

When deriving `palette`:

```ts
const palette = useMemo(() => {
  if (paletteId === 'custom') return buildCustomPalette(customColors);
  return PALETTES.find((p) => p.id === paletteId) ?? PALETTES[0];
}, [paletteId, customColors]);
```

The `'custom'` paletteId is a special-case string — it's NOT in the `PALETTES` array.

### Controls.tsx changes

The palette chips list is extended with a virtual `{ id: 'custom', name: 'Custom' }` entry. When `paletteId === 'custom'`, render 5 color picker rows below the chips.

New props on Controls:
- `customColors: [string,string,string,string,string]`
- `onCustomColorChange: (idx: number, color: string) => void`

### Tests

Add to `tests/data/palettes.test.ts`:
- `buildCustomPalette` returns a Palette with id='custom', the given background, 1 variant, 4 stops
- `buildCustomPalette` assigns colors to the correct stops in order
- `buildCustomPalette` uses alphas `[1.0, 0.85, 0.5, 0.0]` in order
- Result passes the same structural validation as preset palettes (hex regex, monotonic offsets)

## 10. Testing

Following the same strategy as v1 — unit tests for pure logic, manual browser verification for UI.

### Added unit tests

- `tests/renderer/render-helpers.test.ts` — 3 tests for `applyHardness`
- `tests/renderer/composition.test.ts` — 4 new tests (irregularity behavior)
- `tests/data/palettes.test.ts` — 4 new tests for `buildCustomPalette`

Expected total test count after v1.1: 26 (v1) + 11 (v1.1) = 37 passing.

### Manual verification (browser, after implementation)

1. Dureza slider: 0 = blobs look as in v1; 0.5 = visibly tighter/harder; 0.8 = small hard blobs.
2. Irregularidade slider: 0 = round blobs; 0.5 = noticeably bumpy; 1 = clearly torn/organic silhouettes.
3. Custom palette: clicking the chip reveals 5 color pickers with Nightfall defaults; changing any color updates preview within 50ms; Dureza and Irregularidade work on Custom.
4. Existing v1 controls (Randomize, presets, all 3 presets) still work.
5. PNG export from a Custom palette downloads with filename `auragen-custom-{seed}-{W}x{H}.png`.

## 11. Error Handling

No new failure surfaces. Color pickers always return valid hex (browser-enforced). Sliders are bounded. Existing `toBlob` and render error handling stays in place.

## 12. Known Limitations / Future Work

- Custom palette doesn't persist across page reloads (acceptable for this iteration)
- Cluster jitter spawns exactly 4 mini-blobs (no UI control over count — kept simple)
- Hardness compresses all stops uniformly (no per-stop control)
- v2 still pending: WebGL renderer, manual placement, advanced grain
